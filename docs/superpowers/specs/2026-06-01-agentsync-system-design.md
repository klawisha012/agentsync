# Спецификация Системного Дизайна: Рефакторинг и Интеграция Репозиториев AgentSync (2026-06-01)

Эта техническая спецификация описывает переход проекта AgentSync от плоской монолитной структуры к сервисно-слоистой архитектуре (Lean Modular Layout) с обеспечением изоляции домена, разделением пользовательского интерфейса (TUI/Web API) и настройкой локального и удаленного Git-репозиториев.

---

## 1. Цели и Требования к Системному Дизайну

1. **Изоляция доменной логики**: Исключить взаимные неявные зависимости между системной логикой (слияние конфигураций, парсинг) и интерфейсами представления (TUI на Bubbletea, Web HTTP сервер).
2. **Адаптивность под новые ИИ-агенты**: Создать расширяемый интерфейс для интеграции новых целевых ИИ-агентов (Claude Code, Claude Desktop, Antigravity, Cursor и т.д.).
3. **Безопасность конфигураций (Секреты)**: Обеспечить предотвращение случайного коммита файлов секретов (`secrets.env`) и других файлов ИИ-агентов (`.superpowers`, `graphify-out/`).
4. **Портативность и надежность**: Обеспечить работу с портативными изолированными рантаймами (Node.js/Python) без изменения глобальных системных путей пользователя.

---

## 2. Архитектура и Структура модулей (Директории)

Проект реорганизуется в структуру изолированных Go-пакетов внутри директории `internal/`:

```yaml
agentsync/

  │       └── main.go           # CLI-точка входа (парсинг аргументов, запуск TUI, Web или CLI-команд)
  │
  ├── internal/                 # Изолированные пакеты логики
  │   ├── domain/               # Доменные сущности (структуры данных, валидация)
  │   │   ├── types.go          # Базовые общие типы (AgentType, ComponentType и т.д.)
  │   │   ├── manifest.go       # Структуры и валидатор manifest.yaml
  │   │   ├── mcp.go            # Модель спецификации MCP-серверов
  │   │   └── bundle.go         # Модель конфигурационного бандла
  │   │
  │   ├── repository/           # Persistence-слой (Ввод-вывод, Хранилище)
  │   │   ├── store.go          # Чтение/запись канонических файлов манифеста, MCP, правил
  │   │   ├── backup.go         # Управление резервными копиями настроек агентов
  │   │   └── secrets.go        # Парсинг и сохранение secrets.env
  │   │
  │   ├── agent/                # Слой адаптеров для ИИ-агентов
  │   │   ├── adapter.go        # Единый интерфейс AgentAdapter для целевых агентов
  │   │   ├── claude_desktop.go # Адаптер Claude Desktop (слияние config.json)

  │   │
  │   ├── service/              # Сервисный слой (Бизнес-логика)
  │   │   ├── deploy.go         # Smart Merge деплой, фильтрация совместимости и слияние

  │   └── ui/                   # Интерфейсы представления (Presentation Layer)
  │       ├── tui/              # Bubbletea Dashboard
  │       │   ├── model.go      # Модель Bubbletea (цикл Update/View)
  │       │   ├── styles.go     # Стилизация Lipgloss
  │       │   └── view_helpers.go
  │       │
  │       └── web/              # Web-сервер GUI
  │           ├── server.go     # Инициализация HTTP-сервера и роутинг

```

## 3. Слой Адаптеров ИИ-агентов (Интерфейс AgentAdapter)

Вместо жестко прописанной логики деплоя под конкретные агенты, объявляется строгий интерфейс в `internal/agent/adapter.go`:

```go
package agent

import "agentsync/internal/domain"

type AgentAdapter interface {
	// Name возвращает уникальный идентификатор агента (например, "claude-desktop")
	Name() domain.AgentType
	
	// Detect проверяет, установлен ли данный агент на компьютере пользователя
	Detect() bool
	
	// DeployMCP интегрирует настройки MCP-сервера в конфигурационный файл агента
	DeployMCP(mcp domain.MCPConfig, secrets map[string]string) error
	
	// DeployRule записывает правила (Rules) в целевые файлы агента
	DeployRule(rule domain.Rule) error
	
	// ClearMCPs очищает старые канонические MCP-серверы перед обновлением бандла
	ClearMCPs() error
}
```

Все адаптеры реализуют этот контракт, что обеспечивает легкое добавление новых платформ в будущем.

## 4. Конвейер Smart Merge (Деплой и Слияние)

Бизнес-логика деплоя выносится в `internal/service/deploy.go`. Деплой больше не связан с прямым выводом на экран через `fmt.Printf`. Он работает через событийно-ориентированный интерфейс логирования `Logger`:

```go
type Logger interface {
	Log(msg string, level string)
}
```

### Шаги выполнения деплоя:
1. **Загрузка конфигурации**: Чтение манифеста, бандлов, секретов и локальных файлов через `Store`.
2. **Аудит безопасности**: Сканирование на утечку сырых токенов в MCP файлах. Секреты выносятся в `secrets.env`, заменяясь на переменные `${secrets.VAR}`.
3. **Разрешение портативных рантаймов**: Для MCP с указанными полями `runtime` (например, `nodejs@20`) `RuntimeService` находит или устанавливает портативную версию ноды/питона и заменяет команду запуска.
4. **Фильтрация по совместимости**:
   * Для текстовых CLI-агентов (`claude-code`) отсекаются SSE/HTTP MCP-транспорты.
   * Пропускаются плагины, встроенные в агенты нативно (согласно разделу `installed_plugins` в `manifest.yaml`).
5. **Вызов адаптеров**: `DeployService` последовательно вызывает `DeployMCP` и `DeployRule` для каждого активного обнаруженного адаптера.

## 5. Интеграция с Git и Конфигурация Безопасности

Для предотвращения коммита секретов, временных файлов сборки и рабочих пространств ИИ-агентов, в корне проекта создается `.gitignore` со следующей конфигурацией:

```ini

# --- Инструменты ИИ и базы знаний (AI & Agents) ---

frontend/node_modules/

# --- Рантаймы и Портативные пакеты ---
data/runtimes/

# --- Системные файлы и IDE ---
.idea/
.vscode/
*.suo
*.ntvs*
*.njsproj
*.sln
*.sw?
.DS_Store
Thumbs.db
```

### Автоматическое связывание репозиториев:
1. Инициализация локального Git: `git init`.
2. Создание публичного репозитория на GitHub по схеме: `github.create_repository(name: "agentsync", private: false)`.
3. Добавление remote origin в локальный гит.
4. Первичный коммит и отправка (`git push -u origin main`).

## 6. Стратегия Верификации и Тестирования

1. **Модульные тесты (Unit Tests)**:
   * Тесты в `/internal/repository/store_test.go` на корректность сериализации/десериализации YAML/JSON.
   * Тесты слияния MCP-серверов в `/internal/service/deploy_test.go` с использованием mock-логгера.
2. **Ручное тестирование (UI & CLI)**:
   * Запуск `./agentsync.exe web` и проверка открытия дашборда в браузере.
   * Запуск TUI интерфейса через `agentsync.exe tui` и проверка стабильности рендеринга таблиц.
   * Запуск `agentsync.exe doctor` для проведения полной диагностики.
