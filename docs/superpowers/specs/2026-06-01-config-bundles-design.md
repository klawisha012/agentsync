# Спецификация дизайна: Механизм бандлов конфигураций (Config Bundles)

Этот документ описывает создание механизма бандлов конфигураций (Configs) в AgentSync, позволяющего объединять несколько компонентов (MCP, правила, навыки, воркфлоу, хуки) в единые бандлы для синхронизации, удаления и P2P-обмена.

## 1. Архитектура и хранение данных (Backend)

Бандлы будут храниться в папке `configs/` в корне репозитория в формате YAML.
Пример структуры файла `configs/web-dev-bundle.yaml`:

```yaml
name: web-dev-bundle
description: "Инструменты для Fullstack React разработки"
components:
  - type: MCP
    name: StitchMCP
  - type: MCP
    name: github
  - type: Rule
    name: react-style
  - type: Skill
    name: brainstorming
```

### 1.1 Модели данных (`types.go` или `store.go`)
```go
type BundleItem struct {
	Type ComponentType `yaml:"type" json:"type"`
	Name string        `yaml:"name" json:"name"`
}

type ConfigBundle struct {
	ID          string       `json:"id"`
	Name        string       `yaml:"name" json:"name"`
	Description string       `yaml:"description,omitempty" json:"description"`
	Components  []BundleItem `yaml:"components" json:"components"`
}
```

### 1.2 Методы хранилища (`store.go`)
Нам потребуются методы для работы с файлами бандлов:
* `LoadBundles() ([]ConfigBundle, error)` — сканирует папку `configs/*.yaml` и загружает бандлы.
* `SaveBundle(bundle ConfigBundle) error` — сохраняет бандл в `configs/<bundle_id>.yaml`.
* `DeleteBundle(id string) error` — удаляет файл бандла.

### 1.3 HTTP API (`web.go`)
Новые эндпоинты:
* `GET /api/bundles` — получить список всех бандлов.
* `POST /api/bundles` — создать/обновить бандл.
* `DELETE /api/bundles` — удалить бандл.
* `POST /api/bundles/sync` — применить бандл. Синхронизирует все входящие в него компоненты (копирует из глобального в CWD или наоборот, гарантирует их наличие и активность).
* `POST /api/bundles/share` — запустить P2P раздачу для файла бандла.

## 2. Пользовательский интерфейс (Frontend)

Пользовательский интерфейс объединяет в себе селектор бандлов в глобальной шапке и трехпанельный пульт управления:

### 2.1 Глобальная Шапка (Header)
* **Селектор бандла**: Интегрированный выпадающий список для мгновенного выбора активной конфигурации в системе, а также кнопка быстрого создания нового бандла `➕`.
* **Кнопка деплоя**: `Smart Merge Deploy` для запуска автоматического слияния настроек с активными агентами.

### 2.2 Вкладка Agent Manager (Configs)
1. **Левая панель (Категории)**: Список категорий компонентов (`🔌 MCP Серверы`, `📜 Правила`, `🧠 Навыки`, `🔄 Процессы`, `🪝 Хуки`). При клике на категорию обновляется средняя панель.
2. **Средняя панель (Компоненты)**: Список всех доступных компонентов выбранного типа. У каждого компонента отображается название, описание и кнопка `Добавить` / `Убрать` (для быстрого включения/исключения его из активного бандла конфигурации).
3. **Правая панель (Детали бандла)**:
   * **Шапка деталей**: Отображает название и описание активного бандла с кнопкой быстрого удаления `🗑`.
   * **Горизонтальные вкладки содержимого бандла**: Горизонтальный ряд табов для быстрого переключения между типами компонентов, помещенных в бандл, с текстовыми подписями и счетчиками (например, `🔌 MCP (2)`, `📜 Правила (1)`, `🧠 Навыки (1)`, `🔄 Процессы (0)`, `🪝 Хуки (0)`).
   * **Список содержимого**: Отображает компоненты внутри выбранной горизонтальной вкладки с возможностью быстрого удаления (`🗑`).

## 3. План верификации

1. **Создание бандла**:
   * Создать новый бандл `test-bundle` через интерфейс.
   * Убедиться, что файл `configs/test-bundle.yaml` успешно создан с корректной YAML-структурой.
2. **Наполнение бандла**:
   * Добавить MCP `github` и правило `react-style` в бандл.
   * Проверить файл на диске: в нем должны быть прописаны добавленные элементы.
   * Проверить удаление компонента из бандла.
3. **Синхронизация и Шеринг**:
   * Нажать `Синхронизировать` и убедиться, что все файлы бандла активированы.
   * Нажать `Поделиться` и проверить запуск P2P-сервера.
