# Спецификация дизайна: Динамическое определение и настройка Wi-Fi адаптера через Netplan

Эта спецификация описывает переход от жестко заданной конфигурации имени Wi-Fi интерфейса к динамическому определению беспроводного PCI-E модуля на сервере `zwarder-server` с сохранением статического IP-адреса.

## 1. Цели
* **Динамическое определение:** Обеспечить автоматическое применение сетевых настроек к любому подключенному Wi-Fi адаптеру (включая новый Mediatek MT7922 на интерфейсе `wlp1s0`) без необходимости вручную менять его имя в конфигурационных файлах.
* **Сохранение статического IP:** Гарантировать назначение статического IP-адреса `192.168.0.254/24` для беспроводного соединения.
* **Стабильность сети:** Сохранить работоспособность остальных сетевых интерфейсов (`enp2s0`, `wg0` WireGuard-туннеля).

---

## 2. Текущее состояние и проблема
На сервере `zwarder-server` (Ubuntu 24.04, ядро 6.8.0) используется сетевой стек `systemd-networkd`, управляемый через утилиту **Netplan**.

Текущий файл конфигурации `/etc/netplan/01-netcfg.yaml` содержит жесткую привязку к MAC/имени старого Wi-Fi адаптера:
```yaml
  wifis:
    wlx18a6f71bd787:
      dhcp4: false
      addresses:
        - 192.168.0.254/24
      routes:
        - to: default
          via: 192.168.0.1
          metric: 200
      nameservers:
        addresses:
          - 8.8.8.8
          - 1.1.1.1
      access-points:
        "Trisad":
          password: "9657Stop@2"
```

После физического подключения нового PCI-E адаптера `MEDIATEK Corp. MT7922` операционная система инициализировала его под именем **`wlp1s0`**. Поскольку в Netplan имя `wlp1s0` отсутствует, интерфейс не получает настроек и находится в состоянии `DOWN`.

---

## 3. Предлагаемые изменения

### Изменение файла `/etc/netplan/01-netcfg.yaml`
Мы заменим конкретное имя интерфейса `wlx18a6f71bd787` на динамическое сопоставление по шаблону `wl*`. В современном Linux все беспроводные интерфейсы именуются по стандарту *Predictable Network Interface Names* и всегда начинаются с букв `wl`.

#### [MODIFY] `/etc/netplan/01-netcfg.yaml`

Новая структура блока `wifis`:
```yaml
  wifis:
    wifi-dynamic:
      match:
        name: "wl*"
      dhcp4: false
      addresses:
        - 192.168.0.254/24
      routes:
        - to: default
          via: 192.168.0.1
          metric: 200
      nameservers:
        addresses:
          - 8.8.8.8
          - 1.1.1.1
      access-points:
        "Trisad":
          password: "9657Stop@2"
```

---

## 4. План верификации и внедрения

### Этап 1: Резервное копирование
Перед внесением изменений создается резервная копия текущей конфигурации:
```bash
cp /etc/netplan/01-netcfg.yaml /etc/netplan/01-netcfg.yaml.bak
```

### Этап 2: Применение изменений
1. Обновить файл `/etc/netplan/01-netcfg.yaml` в соответствии с дизайном.
2. Проверить синтаксис конфигурации без её постоянного применения с помощью команды `netplan try`. Команда `netplan try` предоставляет встроенный механизм отката изменений (Rollback) через 120 секунд, если связь с сервером пропадет.
3. В случае успешного прохождения проверки применить изменения:
   ```bash
   netplan apply
   ```

### Этап 3: Проверка работоспособности
1. Проверить состояние сетевых интерфейсов:
   ```bash
   ip a show wlp1s0
   ```
   *Ожидаемый результат:* Интерфейс `wlp1s0` находится в состоянии `UP` и имеет IP-адрес `192.168.0.254/24`.
2. Проверить статус беспроводного соединения с точкой доступа "Trisad":
   ```bash
   iwconfig wlp1s0
   # или
   iw dev wlp1s0 link
   ```
   *Ожидаемый результат:* Статус "Connected to Trisad".
3. Убедиться, что доступность по WireGuard-туннелю (`10.8.1.4`) сохраняется.
